## Executive Summary

Wean is a mobile app for habitual Kratom users who want to reduce or quit their intake. It enables systematic, gradual tapering that accounts for Kratom's long half-life, making the process feel achievable rather than punishing. For a substance where cold turkey triggers heroin-like withdrawal, a structured taper plan transforms "quitting" into a series of small wins.

## Functional Requirements

Functional Requirements (FRs) define **what** the system does - specific features and behaviors.

Each requirement includes:
- **ID**: Unique identifier (FR-001, FR-002, etc.)
- **Priority**: Must Have / Should Have / Could Have / Won't Have (MoSCoW)
- **Description**: What the system should do
- **Acceptance Criteria**: How to verify it's complete

---

### FR-001: Quick Dose Logging

**Priority:** Must Have

**Description:**
Users can log a Kratom dose in under 10 seconds with amount, unit selection (grams/ounces), and automatic timestamp capture.

**Acceptance Criteria:**
- [ ] Slider or numeric input for dose amount (0.1g - 50g range)
- [ ] Unit selector toggles between grams (g) and ounces (oz)
- [ ] Timestamp auto-captures current time (user can override)
- [ ] Optional notes field for context
- [ ] Save completes with single tap
- [ ] Haptic feedback confirms save on supported devices
- [ ] Dose appears immediately in daily timeline

**Dependencies:** FR-006 (Authentication)

---

### FR-002: Daily Dose Gauge

**Priority:** Must Have

**Description:**
Visual circular progress indicator showing total daily intake versus target, with color-coded status for at-a-glance assessment.

**Acceptance Criteria:**
- [ ] Circular gauge displays current daily total
- [ ] Numeric display shows exact amount and unit
- [ ] Color coding: Green (under target), Yellow (approaching), Red (over target)
- [ ] Gauge animates smoothly when doses are added
- [ ] Displays "no target set" state gracefully if user hasn't configured goal
- [ ] Tapping gauge navigates to goal settings (premium) or shows upgrade prompt

**Dependencies:** FR-001, FR-009

---

### FR-003: Rolling Calendar Week

**Priority:** Must Have

**Description:**
Horizontal 7-day calendar strip for reviewing historical doses and selecting dates to view.

**Acceptance Criteria:**
- [ ] Displays 7 consecutive days horizontally
- [ ] Current day highlighted visually
- [ ] Days with logged doses show indicator dot or fill
- [ ] Tap day to select and view that day's doses
- [ ] Swipe left/right to navigate to previous/next weeks
- [ ] Shows day of week abbreviation and date number

**Dependencies:** FR-001

---

### FR-004: Daily Dose Timeline

**Priority:** Must Have

**Description:**
Chronological list of all doses for the selected day with ability to edit or delete entries.

**Acceptance Criteria:**
- [ ] Lists all doses for selected date in time order (newest first or oldest first, configurable)
- [ ] Each entry shows: time, amount, unit, notes (if any)
- [ ] Tap entry to edit dose details
- [ ] Swipe or long-press to delete with confirmation dialog
- [ ] Pull-to-refresh updates from server
- [ ] Empty state message when no doses logged for day
- [ ] Running total displayed at top or bottom of list

**Dependencies:** FR-001, FR-003

---

### FR-005: Dark/Light Theme

**Priority:** Must Have

**Description:**
User-selectable theme preference (dark mode or light mode) that persists across sessions.

**Acceptance Criteria:**
- [ ] Settings screen includes theme toggle
- [ ] Dark mode uses dark backgrounds with light text
- [ ] Light mode uses light backgrounds with dark text
- [ ] Theme preference persists to user profile (Firestore)
- [ ] Theme applies immediately without app restart
- [ ] All screens respect theme selection consistently

**Dependencies:** FR-006

---

### FR-006: Anonymous Authentication

**Priority:** Must Have

**Description:**
Firebase-based anonymous authentication that creates user-specific data isolation without requiring email/password.

**Acceptance Criteria:**
- [ ] App automatically creates anonymous user on first launch
- [ ] User ID gates all Firestore data access (per-user collections)
- [ ] No signup form or account creation flow required
- [ ] Auth state persists across app restarts
- [ ] Graceful handling of auth token expiration/refresh
- [ ] User can optionally upgrade to email auth later (premium consideration)

**Dependencies:** None (foundational)

---

### FR-007: Smart Taper Planner

**Priority:** Should Have (Premium)

**Description:**
AI-assisted taper schedule generation based on current usage patterns, desired timeline, and reduction comfort level.

**Acceptance Criteria:**
- [ ] Input wizard collects: current daily average, target daily amount, timeline (weeks)
- [ ] Algorithm generates daily targets using 5-10% weekly reduction (evidence-based)
- [ ] Accounts for Kratom's ~24-hour half-life in reduction curve
- [ ] Displays generated schedule as calendar or list view
- [ ] User can adjust individual day targets manually
- [ ] Plan saves to user profile and updates daily target in gauge
- [ ] Option to regenerate plan if user gets off track
- [ ] Paywall: Requires premium subscription

**Dependencies:** FR-001, FR-002, FR-012

---

### FR-008: Weekly/Monthly Trend Analytics

**Priority:** Should Have (Premium)

**Description:**
Charts and visualizations showing intake trends, averages, and reduction progress over time.

**Acceptance Criteria:**
- [ ] Line chart: Daily totals over past 7/30/90 days
- [ ] Bar chart: Weekly averages comparison
- [ ] Summary stats: Average daily dose, total weekly, % change from prior period
- [ ] Streak counter: Days under target, consecutive reduction days
- [ ] Export chart as image for sharing
- [ ] Paywall: Full analytics require premium; free tier shows 7-day sparkline only

**Dependencies:** FR-001, FR-012

---

### FR-009: Goal Setting & Milestones

**Priority:** Should Have (Premium)

**Description:**
Users can define personal reduction targets and track progress with celebratory milestone achievements.

**Acceptance Criteria:**
- [ ] Set target daily amount (e.g., "5g/day")
- [ ] Set target date (e.g., "by March 31")
- [ ] Progress percentage displayed prominently
- [ ] Milestone celebrations: Badge animations, confetti for hitting targets
- [ ] Milestone examples: "First week under 10g", "7-day streak", "50% reduction achieved"
- [ ] Push notification when milestone unlocked
- [ ] Paywall: Goal setting requires premium; free users see "upgrade to set goals"

**Dependencies:** FR-002, FR-012

---

### FR-010: Dose Reminders & Alerts

**Priority:** Should Have (Premium)

**Description:**
Scheduled push notifications for planned doses and alerts when approaching daily limit.

**Acceptance Criteria:**

- [ ] Alert when over target: "You've exceeded your target for today"
- [ ] User can snooze or dismiss notifications
- [ ] Notification preferences screen to enable/disable types
- [ ] Paywall: Requires premium subscription

**Dependencies:** FR-002, FR-009, FR-012

- [ ] Paywall: Requires premium subscription

## Non-Functional Requirements

Non-Functional Requirements (NFRs) define **how** the system performs - quality attributes and constraints.

---

### NFR-001: Performance

**Priority:** Must Have

**Description:**
App must be fast and responsive to encourage habitual use.

**Acceptance Criteria:**
- [ ] App launch to interactive: <3 seconds on mid-range devices
- [ ] Dose logging flow: <10 seconds end-to-end
- [ ] Screen transitions: <300ms
- [ ] Firestore queries return: <500ms (indexed)
- [ ] Charts render: <1 second for 90-day dataset
- [ ] No UI jank during animations (60fps target)

**Rationale:**
Friction kills habit formation. Users must feel the app is instant.

---

### NFR-002: Security

**Priority:** Must Have

**Description:**
User data must be protected with appropriate security measures.

**Acceptance Criteria:**
- [ ] All data transmitted over HTTPS/TLS
- [ ] Firestore security rules enforce user-only access (uid-gated collections)
- [ ] No sensitive data (health info) stored in logs or analytics
- [ ] Anonymous auth tokens refreshed automatically
- [ ] No third-party analytics SDKs that harvest PII
- [ ] API keys not exposed in client code (environment variables)

**Rationale:**
Substance use data is sensitive. Users must trust their data is private.

---

### NFR-003: Reliability

**Priority:** Must Have

**Description:**
App must be stable and data must never be lost.

**Acceptance Criteria:**
- [ ] 99.9% uptime for Firebase services (Firebase SLA)
- [ ] Zero data loss incidents
- [ ] Graceful offline handling: Queue writes, sync when online
- [ ] Crash-free rate: >99.5%
- [ ] Error monitoring via Sentry or Firebase Crashlytics
- [ ] Automatic retry for failed network requests (exponential backoff)

**Rationale:**
Users rely on accurate historical data for taper plans. Data loss destroys trust.

---

### NFR-004: Usability & Accessibility

**Priority:** Must Have

**Description:**
App must be intuitive and accessible to users with disabilities.

**Acceptance Criteria:**
- [ ] WCAG 2.1 AA compliance for accessibility
- [ ] VoiceOver (iOS) and TalkBack (Android) support
- [ ] Minimum touch target size: 44x44 points
- [ ] Color contrast ratio: 4.5:1 minimum
- [ ] No reliance on color alone for information
- [ ] Support for system font scaling
- [ ] Intuitive navigation: No training required

**Rationale:**
Addiction affects people of all abilities. App must be universally usable.

---

### NFR-005: Scalability

**Priority:** Should Have

**Description:**
Architecture must support growth to 100K+ users without performance degradation.

**Acceptance Criteria:**
- [ ] Firestore indexes optimized for common queries
- [ ] No N+1 query patterns in data fetching
- [ ] Client-side caching reduces redundant reads
- [ ] Firebase costs remain <$500/month at 100K users
- [ ] No single points of failure beyond Firebase infrastructure

**Rationale:**
If product succeeds, growth should not require re-architecture.

---

### NFR-006: Privacy & Compliance

**Priority:** Must Have

**Description:**
App must comply with privacy regulations and respect user data.

**Acceptance Criteria:**
- [ ] Privacy policy published and linked in app
- [ ] Terms of service published and linked in app
- [ ] GDPR compliance: Data export, deletion on request
- [ ] CCPA compliance: Do not sell personal information
- [ ] No social features that risk exposing usage to others
- [ ] Analytics are anonymized (no PII in events)
- [ ] Clear disclaimers: "Not medical advice"

**Rationale:**
Regulatory compliance is required for app store approval and user trust.

---

### NFR-007: Compatibility

**Priority:** Must Have

**Description:**
App must work across target platforms and devices.

**Acceptance Criteria:**
- [ ] iOS 15+ support (covers 95%+ of iOS users)
- [ ] Android 10+ support (API 29+, covers 85%+ of Android users)
- [ ] Web (PWA): Modern browsers (Chrome, Safari, Firefox, Edge)
- [ ] Responsive layout: Phone and tablet form factors
- [ ] Works on devices with limited storage (<100MB install)
- [ ] Handles notch/safe area on modern devices

**Rationale:**
Broad compatibility maximizes addressable market.

## User Stories (High-Level)

User stories follow the format: "As a [user type], I want [goal] so that [benefit]."

These are preliminary stories. Detailed stories will be created in Phase 4 (Implementation).

---

### EPIC-001: Core Tracking Stories

- As a user, I want to log a dose quickly so that tracking doesn't interrupt my day
- As a user, I want to see my doses for any day so that I can review my history
- As a user, I want to edit a dose I entered incorrectly so that my data stays accurate

## User Flows

### Flow 1: First-Time User to First Dose

1. User downloads and opens app
2. Anonymous auth creates account silently
3. Onboarding carousel explains value (3-4 screens)
4. User taps "Log your first dose"
5. Dose entry screen appears with slider at 0
6. User adjusts amount, confirms unit, taps Save
7. Dose appears in timeline, gauge updates
8. Success state: "Great start! Keep tracking."

### Flow 2: Daily Tracking Routine

1. User opens app (returning user)
2. Home screen shows today's gauge and timeline
3. User taps "+" or FAB to add dose
4. Quick entry: Amount slider, tap Save
5. Gauge animates to new total
6. User closes app (10 seconds total)

### Flow 3: Premium Upgrade

1. User taps "View Trends" or "Create Taper Plan"
2. Paywall screen appears with premium benefits
3. User selects monthly or annual plan
4. Native purchase flow (App Store / Play Store)
5. Purchase completes, premium unlocked
6. User accesses requested feature

## Assumptions

**User Assumptions:**
- Users are motivated to reduce/quit and will track consistently if the experience is frictionless
- Users have smartphones (iOS 15+ or Android 10+) with reliable internet
- Users prefer privacy over social accountability features

**Technical Assumptions:**
- Firebase will scale to 100K+ users without issues
- App store approval will not block substance-related tracking app
- React Native/Expo remains a viable framework for 2-3 years

**Business Assumptions:**
- Freemium conversion rate of 5-10% is achievable
- Organic growth through Reddit and SEO can drive initial user base
- Users will pay $9.99/month for premium features that help them quit

## Out of Scope

**Explicitly Excluded from V1:**
- Social features, community forums, or user-to-user messaging
- Direct integration with healthcare systems or EHR platforms
- Medical advice, clinical decision support, or FDA-regulated functionality
- Tracking multiple substances beyond Kratom
- Wearable apps (Apple Watch, Android Wear)
- Offline-first architecture
- Multi-language support (English only)

**Deferred to Future Phases:**
- Accountability partner features (Phase 2)
- Withdrawal symptom tracking (Phase 2)
- Apple Health / Google Fit integration (Phase 2)
- Provider dashboard and B2B features (Phase 3)
